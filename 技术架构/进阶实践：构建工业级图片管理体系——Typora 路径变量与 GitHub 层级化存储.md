# 进阶实践：构建工业级图片管理体系——Typora 路径变量与 GitHub 层级化存储

## 一、 本地侧：开启“一文一仓”隔离模式

在默认设置下，Typora 往往会将所有本地图片散落在同一个目录下 。随着笔记数量的增加，图片会发生严重的“堆叠”和“命名冲突”，导致后期根本无法分清哪张图属于哪篇文章 。

### 1. 核心设置：`${filename}.assets` 变量

- **配置路径**：在 Typora 的 `偏好设置 -> 图像` 中 ，将“插入图片时...”的动作改为 **“复制图片到 ./${filename}.assets 文件夹”** 。

- **具体描述**：当你创建一个名为 `数据结构笔记.md` 的文件时，系统会在同级目录下自动创建一个名为 `数据结构笔记.assets` 的文件夹作为其专属“资产仓” 。

- **工程价值**：这种“一文一仓”的模式实现了本地内容的**物理隔离** 。即使你后期有成百上千篇笔记，每个文档的图片都被锁定在自己的沙盒里，互不干扰，极大地方便了备份和局部迁移 。

## 二、 逻辑侧：基于 YAML 的“公私分流”

对于上传云端的文件及图片，不是所有都会上传的，故需要进行区别 。通过 **YAML 头部命令**，我们可以给每一篇文档装上一个“智能上传开关” 。

![image-20260422160518258](https://cdn.jsdelivr.net/gh/G1234-cod/GYX-BlogImages/img/skill/soft/%E8%BF%9B%E9%98%B6%E5%AE%9E%E8%B7%B5%EF%BC%9A%E6%9E%84%E5%BB%BA%E5%B7%A5%E4%B8%9Aimage-20260422160518258.png)

### 1. 实现“按需上传”的底层逻辑

- **场景 A：私人笔记（默认模式）**：不添加任何命令 。粘贴图片时，Typora 仅将其保存到本地对应的 `.assets` 文件夹中 。这些图片不会触发 PicGo 上传，保证了个人隐私和云端仓库的纯净 。

- **场景 B：博文发布（上传模式）**：在文档开头插入以下三行代码 ：

  ```Markdown
  ---
  typora-copy-images-to: upload
  ---
  ```
  
  一旦检测到该指令，Typora 会自动呼叫 **PicGo** 将图片搬运至 GitHub 云端，并实时将本地路径替换为 CSDN 可直接抓取的网络 URL 。

### 2. 云端纯净度优化：`.gitignore`

为了让 GitHub 云端仓库“看不见”那些琐碎的本地笔记图，我们需要在仓库根目录新建 **`.gitignore`** 文件，并填入 `*.assets/` 规则 。这样，**GitHub Desktop** 在同步时会自动忽略所有本地资产文件夹，确保云端只保留真正需要发布的精华资源 。

## 三、 云端侧：三级目录下的层级化存储

为了让你的 GitHub 仓库不再是一个混乱的“图片堆放场”，我们需要建立一套 **“大类 -> 学科专题 -> 具体文章”** 的层级化存储体系，这样方便后续管理 。

### 1. PicGo 动态路径配置

在开始撰写特定主题的文章前，建议微调 PicGo 的 **“设定存储路径”** ：

- **设定格式**：`img/一级目录/二级目录/文章名/` 。

- **具体案例**：如果你正在写 TCP 相关的博文，路径可设为 `img/Kaoyan/ComputerNetwork/TCP_DeepDive/` 。

- **日期归档技巧**：你甚至可以在路径末尾加上占位符（如 `{year}/{month}/`），让系统按时间维度自动创建文件夹 。

### 2. 避坑指南：路径命名规范

- **严禁使用中文**：虽然 GitHub 支持中文名，但 **jsDelivr CDN** 对中文字符的解析极不稳定，极易导致 URL 编码错误 。

- **建议方案**：一级和二级目录务必使用 **英文或拼音** ，这能确保生成的链接在任何平台（尤其是 CSDN）都能秒开且长久有效 。

------

## 四、 维护侧：不可逾越的“工程底线”

在自动化流水线构建完成后，有一条关于**“持久化链接”**的铁律必须遵守：

- **发布后禁止移动**：图片一旦上传并生成了网络 URL ，你就**绝对不能**在 GitHub 网页端手动修改图片文件名或移动其文件夹位置 。

- **灾难性后果**：任何物理位置的变动都会导致原有的 CDN 链接失效 。这意味着你在 CSDN 或其他平台上发布的文章会瞬间出现大面积“碎图”，且这种损坏是不可逆的 。
